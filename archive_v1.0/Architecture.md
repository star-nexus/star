# 三国演义游戏项目架构

## 架构层级说明

### 1. 配置层 (Config)
- 集中管理游戏各方面的配置参数
- 便于调整游戏平衡性和特性
- 分离硬编码值，提高可维护性

### 2. 引擎层 (Engine)
- 提供游戏运行的基础设施
- 管理游戏状态和生命周期
- 处理核心事件系统
- 管理游戏资源

### 3. 领域层 (Domains)
- **世界领域 (World)**: 处理地图生成、地形、气候等
- **角色领域 (Characters)**: 管理武将、军队等实体
- **战斗领域 (Combat)**: 处理战斗规则和机制
- **外交领域 (Diplomacy)**: 处理势力间关系
- **经济领域 (Economy)**: 管理资源和经济系统

### 4. 服务层 (Services)
- **AI服务 (AIService)**: 提供各种AI系统，包括LLM集成
- **叙事服务 (NarrativeService)**: 处理故事和事件系统
- **持久化服务 (PersistenceService)**: 管理游戏存档和加载

### 5. 接口层 (Interface)
- **UI**: 游戏界面组件
- **场景 (Scenes)**: 游戏各场景管理
- **输入 (Input)**: 处理用户输入
- **渲染 (Rendering)**: 负责游戏画面渲染

### 6. 工具层 (Utils)
- 提供通用工具函数和类
- 数学工具、日志系统、调试工具等

### 7. 资源层 (Assets)
- 管理游戏素材资源
- 图像、音频、数据文件等

### 8. 测试层 (Tests)
- 单元测试和集成测试
- 性能测试和AI行为测试

## 架构设计理由

1. **领域驱动设计**
   - 按游戏领域而非技术层次划分，符合三国题材游戏的复杂度
   - 每个领域内高内聚，领域间低耦合

2. **分层架构**
   - 清晰的依赖方向：上层依赖下层，不允许循环依赖
   - 每层有明确定义的职责范围

3. **模块化设计**
   - 每个模块都可独立开发和测试
   - 便于团队协作和功能扩展

4. **服务隔离**
   - AI和叙事系统作为服务提供，与核心游戏逻辑解耦
   - 便于替换或升级LLM模型而不影响游戏核心

5. **统一事件系统**
   - 通过事件系统进行模块间通信
   - 减少直接依赖，提高系统弹性

## 关键模块交互

1. **地图生成与战斗系统**
   - 地图生成器创建地形
   - 地形影响战斗规则
   - 战斗结果可能改变地形

2. **AI与游戏世界**
   - AI服务解析世界状态
   - 基于LLM做出决策
   - 通过角色和军队实体执行决策

3. **叙事与游戏机制**
   - 叙事服务触发事件
   - 事件影响世界和角色状态
   - 玩家决策反过来影响叙事发展

4. **UI与游戏领域**
   - UI层展示各领域信息
   - 用户输入转化为领域行为
   - 领域状态变化反馈到UI

## 开发优势

1. **增量开发友好**
   - 可以先实现核心领域，再逐步添加服务和高级功能
   - 每个模块可独立迭代

2. **技术选型灵活**
   - 领域和服务的分离允许不同模块使用最适合的技术
   - LLM集成模块可以独立演化

3. **测试便利性**
   - 模块间清晰的边界便于单元测试
   - 可以模拟外部依赖进行隔离测试

4. **维护性和可扩展性**
   - 良好的解耦设计降低了修改成本
   - 新功能可以以新的领域或服务形式添加
